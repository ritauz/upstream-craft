# システム方式設計

## 🎯 目的
要件定義で合意した要件を、**全体システムの方式（アーキテクチャ・構成・外部連携）** に落とし込み、基本設計の土台を固める。  
システムの大枠を定義することで、開発方針・利用技術・インフラ要件を明確にする。

---

## 🤝 顧客との進め方

### 1. 準備フェーズ
- 要件定義書（業務要件・非機能要件・制約条件）を確認  
- 想定するシステムアーキテクチャのパターン（オンプレ／クラウド、Web3層構造、API連携方式など）を整理  

### 2. 顧客ヒアリング
- 業務特性や既存環境を踏まえ、システム方式の方向性を確認  
- ヒアリング観点：
  - システムの構成方式（集中型／分散型／クラウド利用）  
  - 外部システムとの連携方式（API／バッチ／ファイル連携）  
  - 運用方式（監視・バックアップ・リリース管理）  

### 3. テンプレ初期入力
- [システム方式設計書](/docs/design/templates/system-architecture) に方式・構成図を反映  
- 不明点は「未確認」として残す  

### 4. レビュー・合意形成
- 顧客と構成案を確認し、運用や制約との齟齬がないかチェック  
- 合意した方式案を「承認済」とする  

### 5. 継続的な更新
- 新たな要件変更や制約が出た場合は即時反映  
- 更新履歴を残し、次工程（機能設計・データ設計）に影響を伝播  

---

## 📥 入力となる成果物
- 要件定義書  
- 非機能要件一覧  
- 制約条件・適用範囲  

## 📤 出力となる成果物
- [システム方式設計書](/docs/design/templates/system-architecture)  

---

## ✅ 最終的に目指す状態
- システム全体方式（アーキテクチャ、外部連携、運用方針）が顧客と合意済み  
- 次工程の詳細設計・実装にブレなく進める状態
