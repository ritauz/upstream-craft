# 要件定義工程の全体像

要件定義工程は、システム開発における **出発点** であり、以降の設計・実装・テストの前提を決定します。  
顧客と開発チームが共通理解を持ち、プロジェクトのスコープを明確にすることが目的です。

---

## 🚀 進め方の流れ

1. **業務要件の明確化**  
   - 目的・KPI・アクター・トリガー・前提条件を整理し、業務プロセスを言語化する。  

2. **機能要件の整理**  
   - 業務フローやユースケースをベースに、必要な機能と入出力を定義する。  

3. **ユースケース図・業務フロー図の作成**  
   - 視覚化により顧客との認識齟齬を減らす。  

4. **非機能要件の合意形成**  
   - 可用性、性能、運用、セキュリティ、移行性、環境制約などを IPA 非機能要求グレードに基づきレベル化する。  

5. **制約条件・適用範囲の確認**  
   - 法規制、対象範囲、システム境界を定め、除外事項を明記する。  

---

## 📑 プロセスと成果物の対応

| プロセス（やること） | 成果物（アウトプット） |
| --- | --- |
| 業務要件の明確化 | **業務要件一覧書**（目的、KPI、業務プロセス、アクター、前提条件、アウトカム） |
| 機能要件の整理 | **機能要件一覧書**（機能ID、名称、説明、入出力、優先度） |
| ユースケース図／業務フロー図の作成 | **ユースケース図／業務フロー図** |
| 非機能要件の合意 | **非機能要件定義書**（稼働率、RPO/RTO、レスポンス時間、ユーザ数、バックアップ方式 等） |
| 制約条件・適用範囲の確認 | **要件定義書**（範囲・境界・除外事項を明記） |
